# Specification: простой AI-агент с локальной LLM (Ollama + qwen3.5:4b)

## 1. Цель

Разработать простого, но полноценно работающего AI-агента на Python, который:

- принимает задачу от пользователя;
- анализирует её с помощью локальной LLM;
- умеет использовать инструменты;
- работает в цикле `thought → action → observation → thought → ...`;
- самостоятельно принимает решение, какой следующий шаг нужен;
- завершает работу финальным ответом.

Агент должен быть минималистичным по объёму, но архитектурно аккуратным, расширяемым и пригодным для дальнейшего развития.

---

## 2. Основные требования

### 2.1. Локальная модель

Обязательно использовать локальную модель:

- `qwen3.5:4b`

Через:

- `ollama`

### 2.2. Формат работы агента

Агент должен работать в цикле:

1. Получить пользовательскую задачу.
2. Передать в модель контекст:
   - системную инструкцию;
   - описание доступных инструментов;
   - историю шагов;
   - текущую задачу.
3. Получить от модели структурированный ответ.
4. Если модель вернула действие:
   - вызвать соответствующий инструмент;
   - сохранить результат как `observation`;
   - перейти к следующему шагу цикла.
5. Если модель вернула финальный ответ:
   - завершить цикл;
   - вернуть результат пользователю.

### 2.3. Структурированный формат ответа модели

Модель должна возвращать строго один из двух JSON-форматов.

#### Вариант 1. Действие

```json
{
  "thought": "Нужно использовать калькулятор для вычисления выражения",
  "action": "calculator",
  "args": {
    "expression": "(123 + 456) * 2"
  }
}
```

#### Вариант 2. Финальный ответ

```json
{
  "final_answer": "Результат вычисления: 1158"
}
```

### 2.4. Контроль выполнения

Нужно реализовать:

- ограничение количества шагов, например `maxSteps = 8`;
- защиту от бесконечного цикла;
- завершение с ошибкой, если модель:
  - вернула некорректный JSON;
  - указала неизвестный инструмент;
  - превысила число шагов;
  - вернула невалидную структуру.

## 3. Сценарий работы агента

## 3.1. Общая схема

```text
User Task
   ↓
Agent Runner
   ↓
Build Prompt
   ↓
LLM (Ollama / qwen3.5:4b)
   ↓
Parse JSON response
   ↓
if action:
    execute tool
    save observation
    continue loop
if final_answer:
    finish
```

## 3.2. Внутренний цикл

Каждый шаг цикла должен содержать:

- `stepIndex`
- `thought`
- `action` или `finalAnswer`
- `observation` (если был вызван инструмент)

Пример логики:

1. Модель думает:
   - “Нужно прочитать файл”.
2. Агент вызывает инструмент `readFile`.
3. Получает `observation`.
4. На следующем шаге модель использует `observation`.
5. Затем либо вызывает ещё один инструмент, либо завершает задачу.

---

## 4. Инструменты

На первом этапе достаточно реализовать минимум 3 инструмента.

## 4.1. Calculator tool

Назначение:

- вычисление простых арифметических выражений.

Пример вызова:

```json
{
  "action": "calculator",
  "args": {
    "expression": "(123 + 456) * 2"
  }
}
```

Пример результата:

```json
{
  "ok": true,
  "result": 1158
}
```

Требования:

- поддерживать `+ - * / ( )`;
- не использовать `eval` в небезопасном виде;
- либо реализовать безопасный парсер выражений, либо использовать ограничённый безопасный механизм.

---

## 4.2. Read file tool

Назначение:

- чтение текстового файла.

Пример вызова:

```json
{
  "action": "readFile",
  "args": {
    "path": "data/input.txt"
  }
}
```

Пример результата:

```json
{
  "ok": true,
  "content": "line1\nline2\nline3"
}
```

Требования:

- читать только текстовые файлы;
- предусмотреть обработку ошибок:
  - файл не найден;
  - нет доступа;
  - файл слишком большой;
  - неверная кодировка.

Дополнительно допустимо ограничить размер файла, например:

- не более 1 MB.

---

## 4.3. HTTP request tool

Назначение:

- выполнять HTTP GET-запросы.

Пример вызова:

```json
{
  "action": "httpGet",
  "args": {
    "url": "https://example.com"
  }
}
```

Пример результата:

```json
{
  "ok": true,
  "statusCode": 200,
  "text": "..."
}
```

Требования:

- поддерживать только `GET` на первом этапе;
- устанавливать timeout;
- обрабатывать ошибки сети;
- ограничивать размер возвращаемого текста;
- по возможности возвращать укороченный текст, например первые 5000 символов.

---

## 4.4. Optional: simple search tool

Если будет реализован простой поиск, можно сделать инструмент:

- `simpleSearch`

Варианты реализации:

- поиск по локальному набору данных;
- поиск по файлам;
- обёртка над HTTP API.

Но для первой рабочей версии это не обязательно, если уже есть `calculator`, `readFile`, `httpGet`.

## 6. Требования к архитектурным компонентам

## 6.1. LLM client

Компонент отвечает за вызов Ollama.

Обязанности:

- отправить prompt в Ollama;
- получить текст ответа;
- вернуть его в сервис разбора ответа.

Интерфейс должен быть абстрагирован через protocol.

Пример ответственности:

- `LlmClientProtocol`
- `OllamaClient`

---

## 6.2. Prompt builder

Компонент собирает prompt для модели.

В prompt обязательно должно входить:

- роль агента;
- список доступных инструментов;
- правила возврата только JSON;
- история предыдущих шагов;
- текущая пользовательская задача.

Prompt должен жёстко требовать:

- вернуть только валидный JSON;
- не писать пояснительный текст вне JSON;
- выбирать только доступные инструменты.

---

## 6.3. Response parser

Компонент отвечает за:

- разбор JSON-ответа от модели;
- валидацию структуры;
- определение:
  - это `action`;
  - или это `final_answer`.

Если JSON невалиден, агент должен либо:

- сделать controlled failure;
- либо, опционально, один раз попросить модель вернуть корректный JSON.

Для первой версии допустимо просто завершать задачу с понятной ошибкой.

---

## 6.4. Tool registry

Компонент хранит зарегистрированные инструменты и позволяет:

- получить инструмент по имени;
- проверить, существует ли он;
- вызвать его.

---

## 6.5. Agent runner

Это главный orchestration-компонент.

Обязанности:

- принять задачу;
- инициализировать цикл;
- на каждом шаге:
  - построить prompt;
  - вызвать LLM;
  - распарсить ответ;
  - при необходимости вызвать tool;
  - сохранить observation;
  - продолжить цикл;
- остановить выполнение по:
  - `final_answer`;
  - превышению лимита шагов;
  - ошибке.

---

## 7. Требования к prompt

Системная инструкция модели должна содержать примерно такую логику:

- Ты AI-агент, работающий пошагово.
- У тебя есть инструменты.
- На каждом шаге ты обязан вернуть только один JSON.
- Ты можешь:
  - либо выбрать действие;
  - либо вернуть финальный ответ.
- Если для решения нужен инструмент — используй его.
- Если уже достаточно информации — верни `final_answer`.
- Не добавляй markdown.
- Не добавляй пояснения вне JSON.

### 7.1. В prompt нужно перечислить доступные инструменты

Например:

```text
Available tools:

1. calculator
Args:
{
  "expression": "string"
}

2. readFile
Args:
{
  "path": "string"
}

3. httpGet
Args:
{
  "url": "string"
}
```

### 7.2. В prompt нужно добавлять историю шагов

Например:

```text
Previous steps:

Step 1:
Thought: Нужно прочитать файл
Action: readFile
Args: {"path": "data.txt"}
Observation: {"ok": true, "content": "line1\nline2"}
```

## 8. Логирование и трассировка

Агент должен показывать ход работы.

Минимум нужно логировать:

- пользовательскую задачу;
- номер шага;
- `thought`;
- вызванный `action`;
- `args`;
- `observation`;
- финальный ответ;
- ошибки.

Пример вывода в консоль:

```text
User task: Прочитай файл и скажи сколько там строк

Step 1
Thought: Нужно сначала прочитать содержимое файла
Action: readFile
Args: {"path": "data/input.txt"}
Observation: {"ok": true, "content": "a\nb\nc"}

Step 2
Thought: Теперь можно посчитать количество строк
Final answer: В файле 3 строки
```

---

## 9. Ограничения безопасности и устойчивости

## 9.1. Ограничение шагов

Обязательно:

- `maxSteps` в конфиге, например `8`.

Если лимит исчерпан, вернуть ошибку вида:

```text
Agent stopped: maximum number of steps exceeded
```

## 9.2. Ограничение инструментов

Нельзя позволять модели вызывать произвольный код.

Можно вызывать только те инструменты, которые зарегистрированы в `toolRegistry`.

## 9.3. Ограничение файлового доступа

Для `readFile` желательно ограничить чтение:

- только из разрешённой директории, например `workspace/`.

## 9.4. Ограничение HTTP

Для `httpGet` обязательно:

- timeout;
- ограничение размера ответа;
- обработка сетевых ошибок.

## 10.1. Язык и версия

## 10.3. OOP и расширяемость

## 11. Модели данных

Ниже пример сущностей, которые стоит реализовать.

## 11.1. AgentStep

```python
class AgentStep(BaseModel):
    stepIndex: int
    thought: str | None
    action: str | None
    args: dict[str, Any] | None

```

## 11.2. ActionResponse

```python
class ActionResponse(BaseModel):
    thought: str
    action: str
    args: dict[str, Any]
```

## 11.3. FinalResponse

```python
class FinalResponse(BaseModel):
    finalAnswer: str
```

## 12. Protocol-интерфейсы

Нужно использовать `typing.Protocol`.

## 12.1. LlmClientProtocol

```python
class LlmClientProtocol(Protocol):
    def generate(self, in_prompt: str) -> str:
        ...
```

## 12.2. ToolProtocol

```python
class ToolProtocol(Protocol):
    @property
    def name(self) -> str:
        ...

    def execute(self, in_args: dict[str, Any]) -> dict[str, Any]:
        ...
```

## 14. Требования к Ollama integration

Агент должен работать с локально поднятым Ollama.

Перед запуском должно подразумеваться, что пользователь выполнил:

```bash
ollama pull qwen3.5:4b
ollama run qwen3.5:4b
```

Либо агент сам делает запрос к локальному API Ollama.

Допустимы 2 способа интеграции:

1. Через Python-библиотеку.
2. Через HTTP API Ollama.

Предпочтительно:

- сделать адаптер так, чтобы реализация могла быть заменена без изменения `agentRunner`.

## 17. Нефункциональные требования

## 17.1. Надёжность

Код не должен падать без понятного сообщения.

На любую ожидаемую ошибку должен быть контролируемый ответ.

## 17.2. Расширяемость

Должно быть легко добавить новый инструмент без переписывания основного цикла.

Ожидаемый путь расширения:

- создать новый класс tool;
- зарегистрировать его в `toolRegistry`;
- обновить prompt builder.

## 17.3. Простота

Первая версия должна быть максимально простой, но не одноразовой.

Нельзя делать “грязный прототип”, который невозможно расширять.

- журнал шагов в отдельный файл;
- streaming ответа модели;
- summarization длинных observations.

Но всё это не входит в MVP.

## 24. Итог

Нужно получить минимального, но настоящего AI-агента, который:

- принимает задачу;
- думает пошагово;
- вызывает инструменты;
- использует результаты инструментов;
- сам решает, что делать дальше;
- ограничен по числу шагов;
- безопасно завершает выполнение;
- работает на локальной модели `qwen3.5:4b` через `ollama`.

Это должен быть не просто чат с моделью, а базовый agent loop с tool use и контролем исполнения.
